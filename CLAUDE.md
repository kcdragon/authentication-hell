# Authentication Hell

A browser-based game that satirizes the daily pain of workplace authentication and 2FA. Players endure increasingly absurd login, verification, and MFA flows — the real enemy isn't a boss, it's SSO.

The project started from a Rails + Inertia + React + shadcn starter kit and is being reshaped around the game's theme. Game mechanics are not yet designed; the current work is reskinning the starter kit and establishing the visual/voice foundation before the first minigame lands.

## Theme & voice

- **Tone:** satirical, mournful, deadpan. Framing frustrating auth flows as literal suffering.
- **Recurring motifs:** "suffering," "descent," "Hell," "carrier pigeon," absurd MFA method counts. Reuse sparingly — not every string needs to be a joke.
- **Example copy already live:** homepage tagline "Login. Verify. Captcha. Repeat. Win?"; sign-in header "Welcome back to Hell"; sign-up header "Begin the descent"; footer "By continuing, you consent to receive 2FA codes via carrier pigeon."

## Visual language

- Dark-first palette with a red `destructive` accent. Radial red glow overlay at the top of page shells.
- Prefer Tailwind design tokens (`bg-background`, `text-foreground`, `text-destructive`, `bg-card`, `text-muted-foreground`) over hardcoded hex — gives automatic dark-mode support.
- Eyebrow pills (small, uppercase, tracked, destructive-tinted) to set page mood before the H1.
- Bold display titles (`text-5xl`–`text-7xl`, tracking-tight). Wrap "Hell" in `text-destructive` when it appears in a title.
- Reuse shadcn/ui primitives (`@/components/ui/*`); wrap Inertia `<Link>` inside `<Button asChild>` for themed links.

## Architecture

**Stack:** Rails 8.1 + Inertia.js + React 19 + TypeScript + Tailwind CSS 4 + shadcn/ui + SQLite + `authentication-zero`. Ruby 4.0.2.

### Request Flow

Rails handles routing and data fetching. Inertia renders React components server-side by name — no REST API layer. Controllers pass props directly to React page components. The bridge is `InertiaController`, which app controllers inherit from.

### Controller Hierarchy

```
ApplicationController       # before_action :authenticate, Current context
  └── InertiaController     # inertia_config, inertia_share (auth)
        ├── HomeController (skips :authenticate, still calls perform_authentication)
        ├── DashboardController
        ├── SessionsController, UsersController
        ├── Identity::EmailVerificationsController, Identity::PasswordResetsController
        └── Settings::* (profile, password, email, sessions, appearance)
```

`inertia_share` in `InertiaController` provides `auth: { user, session }` to every page.

### Frontend Structure

```
app/frontend/
  entrypoints/inertia.tsx   # Vite entry point, Inertia app init
  pages/                    # React page components (matched by Inertia to controller renders)
  layouts/                  # app-layout (authed), auth-layout (sign in/up), persistent-layout, settings/
  components/               # Reusable components; ui/ contains shadcn primitives
  hooks/                    # Custom React hooks
  types/index.ts            # Shared TypeScript interfaces (SharedProps, Auth, User, Session, NavItem, BreadcrumbItem, FlashData)
  routes/index.js           # Auto-generated by js-routes — do not edit
```

Inertia resolves page components by path: `render inertia: "home/index"` → `pages/home/index.tsx`.

### Route Helpers

`js-routes` auto-generates `app/frontend/routes/index.js` (and `.d.ts`) from Rails routes (camelCase). In development the file is regenerated when routes change. Import and use like: `import { signInPath, signUpPath, dashboardPath } from "@/routes"`.

### TypeScript Types

Shared types live in `app/frontend/types/index.ts`: `SharedProps`, `Auth`, `User`, `Session`, `NavItem`, `BreadcrumbItem`, `FlashData`. Import from `@/types` — do not redeclare locally in page files. Access shared props via `usePage<SharedProps>()`.

### Domain Model

- **User** — `has_secure_password`, session tokens, email verification, password reset. Password minimum: **8 characters** (`app/models/user.rb`).
- **Session** — belongs to User; stores `user_agent`, `ip_address`, cookie-backed auth.

Game models (players, runs, minigames, scores) are not yet designed.

### Authentication

Session-based via `authentication-zero`. `ApplicationController#authenticate` runs on every request, reads `cookies.signed[:session_token]`, and sets `Current.session`. Unauthenticated requests redirect to `sign_in_path`. `Current.user` / `Current.session` (see `app/models/current.rb`) expose the identity layer. Auth data is shared to Inertia via `inertia_share auth:`.

Controllers that should be reachable logged-out (e.g. `HomeController`) use `skip_before_action :authenticate` but still call `perform_authentication` so the page can branch on auth state.

## Testing

Write RSpec tests for all backend code changes. Tests go in `spec/` mirroring the app structure:
- Models: `spec/models/`
- Controllers: `spec/requests/`
- Jobs: `spec/jobs/`
- Mailers: `spec/mailers/`

Use FactoryBot for test data (`spec/factories/`). The `AuthenticationHelpers` support module is auto-included in `:system`, `:request`, and `:controller` specs. Run the relevant spec file after writing tests to confirm they pass before running `bin/ci`.

After implementing a plan, run `bin/ci` and fix any issues before considering the work complete. Use WebMock or VCR to test external API calls.

### What to test

Write tests for every model, controller, job, service, and serializer you add or modify. Keep tests focused and simple — cover the main happy path and any significant error cases (validation failures, authorization checks, job side effects, serialized field shaping). Skip exhaustive guard-clause coverage; test behavior that matters, not every branch.

- **Models:** validations, associations, scopes, and non-trivial methods
- **Controllers (request specs):** that the correct Inertia component is rendered, key props are present, and significant failure paths (e.g. unauthenticated access redirects)
- **Jobs:** that the job performs its primary side effect
- **Serializers:** key field shaping, conditional fields, derived URLs or nested payloads

The `inertia_rails/rspec` matchers (`be_inertia_response`, `render_component`, `have_props`, `have_flash`, etc.) are not wired into `spec/rails_helper.rb` yet. Add `require "inertia_rails/rspec"` there if/when you start writing Inertia-aware request specs.

## Conventions

- **Ruby:** frozen string literals, double quotes, rubocop-rails-omakase style. Ruby 4.0.2 (managed by rv).
- **Migrations:** create model-backed table migrations with `bin/rails generate model ...`; do not hand-write multiple table creations into a single migration.
- **Migration edits:** never edit an existing migration unless you first roll it back (or otherwise revert DB state) so the schema history stays consistent.
- **Seeds:** seed data must be idempotent — rerunning `bin/rails db:seed` should reuse or update existing records instead of creating duplicates.
- **React:** functional components, `usePage<SharedProps>()` to access shared Inertia props.
- **Internal links:** use Inertia's `<Link>` (from `@inertiajs/react`), not plain `<a>` tags, for client-side navigation. For themed buttons, wrap as `<Button asChild><Link href={…}>…</Link></Button>`.
- **Forms:** use Inertia's `useForm` with data nested under the resource key (e.g. `{ user: { email: "" } }`) to match `params.require(:user).permit(...)` on the Rails side.
- **shadcn/ui:** components live in `app/frontend/components/ui/` — add new ones with `npx shadcn@latest add <component>`.
- **Styling:** prefer Tailwind design tokens over hardcoded hex. Don't introduce styled `<div>`s where a shadcn primitive exists.
- **Copy:** keep voice-setting copy in page components, not the layout, so pages can tailor their tone. The auth layout accepts an optional `eyebrow` prop for the pill.
- **Reuse:** avoid duplicating logic across React components — extract shared logic into a reusable component and have both call it.
